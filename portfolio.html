ApexFinance — Personal Finance SPA

● Live
JavaScript ES6+ HTML5 Vanilla CSS Chart.js LocalStorage API

Problem Statement

Most personal finance tools are either bloated SaaS subscriptions or primitive spreadsheets. I needed a zero-dependency, client-side-only solution that demonstrates full SPA architecture — state management, reactive rendering, and persistent data — without a backend. The real problem was engineering correctness: how do you keep UI, data, and computed metrics in sync with zero framework overhead?

System Architecture & Tech Stack

State Layer Single source of truth array in localStorage, serialized as JSON on every mutation
Rendering Custom reactive render loop — all UI derived from state, no direct DOM mutations on user events
View Routing Hash-based SPA routing with history API; two independent view panels (Finance + Portfolio)
Visualization Chart.js doughnut with dynamic dataset updates; graceful empty-state handling

Key Technical Challenges & Optimization

  • Performance Replaced full ledger re-render on every keystroke with debounced filter computation (300ms delay) — reduced DOM operations by ~80% during live search.
  • State Integrity Introduced UUID-based transaction IDs to decouple array indices from entity identity, eliminating off-by-one errors during CRUD operations on filtered views.
  • Data Validation Implemented client-side schema guards on all localStorage reads to prevent runtime crashes from malformed data (e.g., corrupted entries from browser extensions).
  • CSV Export Built a pure-JS CSV serializer that handles comma-escaping edge cases in description fields — output is Excel-compatible without a library dependency.

Future Iterations

IndexedDB migration for structured querying and transaction-safe writes
Firebase Firestore sync for cross-device state persistence
ML-based spend categorization using a lightweight on-device classifier
Budget threshold alerts via Web Push Notifications API

StudyOS — CLI Study Session Tracker in C

◐ WIP
C Language File I/O Structs & Pointers Dynamic Memory Linux CLI

Problem Statement

Digital study apps track time but never analyze efficiency. I wanted to build a systems-level tool that records study sessions, computes rolling averages of productive hours, and surfaces insights — all from the terminal. The constraint: implement it entirely in C, using manual memory management and flat-file persistence. This forces mastery of pointers, structs, and file I/O that higher-level languages abstract away.

System Architecture & Tech Stack

Data Model struct Session with subject, duration, timestamp, and productivity score fields
Persistence Binary flat-file (sessions.dat) using fwrite/fread for O(1) append and full sequential scan
Memory Dynamic array via malloc/realloc — doubling growth strategy to minimize heap reallocations
Interface ANSI-escape-code terminal UI with colored output, progress bars, and formatted tabular reports

Key Technical Challenges & Optimization

  • Memory Safety Implemented a custom bounds-checked array wrapper to catch buffer overflows — caught 3 silent memory corruption bugs that would have been invisible in Python.
  • Complexity Optimized weekly-average computation from O(N²) naive double-loop to O(N) single-pass using a sliding accumulator — critical for large session histories.
  • Sorting Implemented in-place qsort with a custom comparator for multi-field sort (primary: date, secondary: subject name) without allocating a secondary buffer.
  • Edge Cases Handled file corruption gracefully: magic-number header validation rejects malformed .dat files before heap allocation occurs.

Future Iterations

Unix socket IPC for a companion background daemon to track idle time
ASCII chart rendering for weekly productivity heatmaps in-terminal
JSON export mode for data ingestion into external dashboards
Multi-user mode with file-level locking (flock) for concurrent access
